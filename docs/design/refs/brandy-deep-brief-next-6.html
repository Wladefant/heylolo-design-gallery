1. Delete Account flow polish — #215

What it is

Apple's App Store guideline 5.1.1(v) says: any app that lets you create an account must let you delete that account from inside the app — not via a web link, not via "email us." We already let a parent delete a single child profile, but that is not the same thing. Reviewers specifically hunt for whole-account deletion, and a web-only deletion path fails. So this is a binary gate for the Jul 20 freeze — it must be in the build we submit, and it must be honest and complete.

Functionally, "delete my account" is the nuclear option. It's the parent saying "erase everything you hold about my family." When they confirm, the backend cascades: for every child under that parent it deletes all conversations, all messages, the daily digest, the memory profile, day-aggregates, mood reports, any generated images in Storage, the child document, and the child's own Firebase Auth login. Then it deletes the parent's own documents (profile, identities, subscription/entitlement record, device/notification tokens). And only at the very end does it delete the parent's Auth user — the login itself. That ordering is deliberate: if anything fails partway through, the parent's login still exists, so they can sign back in and press delete again and it retries cleanly. Delete the login first and a half-finished failure would orphan data the parent can no longer reach. "Retryable by construction" is the whole reason the parent Auth user goes last.

This was just implemented dev-first on branch feat/account-deletion-fe — meaning the plumbing works but it wears no design. Today it is: a red "Delete Account" row at the bottom of Settings that reuses the Log Out icon (placeholder — it's literally LogoutIcon tinted coral), and a plain confirmation screen with a title, a paragraph, and a text box where the parent must type the word DELETE before the button enables. It works; it just doesn't yet feel like the grave, honest moment it is. That's your job here.

How it's planned technically (what exists vs what you're shaping)

  • Already built and wired (I read the branch): a destructive Settings row → a dedicated confirmation page (delete-account-page.tsx) → a type-DELETE-to-confirm gate → a call to the backend v1_deleteParentAccount cascade → local sign-out → return to the welcome/auth screen. In-flight the button shows a spinner and is disabled; on failure it shows a retryable alert ("Couldn't delete account — check your connection and try again").
  • The confirmation gate is type-DELETE, not a PIN. The original issue floated re-entering the security PIN, but the Security screen only sets a PIN, it never verifies one (there's no server verify endpoint), so the team went with the Apple-standard typed-word confirmation instead. Design around typing the word DELETE, not a PIN pad.
  • The destructive tint already exists as a token: the row and button use Palette.CoralRed. There's a proper destructive Button variant. So you're not inventing a color system — you're art-directing within the existing warm-teal, light-only palette, with coral as the "this is serious" accent.
  • What's placeholder and needs you: (a) the icon — a reused logout glyph, wrong metaphor; (b) the confirmation screen's whole tone and composition — right now it's a bare title + paragraph + input; (c) the "what will be deleted" list — currently one run-on sentence ("every child profile, all of their conversations, memories, and settings"), which should become a scannable, honest list so the parent sees exactly what evaporates; (d) the in-flight and failure states as designed moments, not a default spinner and a system alert.

Your design task (screens + states)

  1. A destructive icon for the Settings row — a proper "delete account" glyph (not a door/logout, not a scary trash-fire). It should read as serious and final without being alarming. Fits the existing Settings row layout (icon left, title + subtitle, chevron/press).
  2. The confirmation screen — the emotional center. A parent doing something serious and irreversible. Tone: grave, honest, respectful — never scary, never a horror-movie red wall. Think "a serious document you're signing," not "a self-destruct button." Warm even here.
  3. The "what will be deleted" list — a clear, honest, scannable enumeration of what goes: every child and their conversations/memories/images, your account, your subscription. Honesty is the design; the parent should feel we're being transparent, not hiding the blast radius. Consider how it reads for a parent with 1 child vs 3 children.
  4. The type-DELETE confirm control — the input + the "type DELETE to confirm" prompt, and the primary destructive button (disabled until the word matches, enabled state, loading state).
  5. In-flight state — while the cascade runs (it can take a few seconds across many children): a calm, non-frightening "deleting your account…" state. Not a panic spinner.
  6. Failure state — a gentle, retryable message ("something went wrong, you can try again" — the account is still deletable on retry by design). Not a system alert box.
  7. The exit — after success the app signs out and returns to the welcome screen. Consider whether there's a brief, dignified "your account has been deleted" acknowledgment before that, or a clean hand-off. (No confirmation email is part of this flow — it's in-app only.)

Hard constraints

  • Light-only. Warm tide-pool palette; coral (Palette.CoralRed) is the destructive accent. No dark mode.
  • Grave but not scary. This is a premium, trustworthy brand doing a hard thing honestly. No blood-red full-bleed, no skull iconography, no dark-pattern friction that tries to talk the parent out of it beyond the single honest type-to-confirm gate.
  • Honesty is the brand. The "what gets deleted" list should over-disclose, not under-disclose. A parent should trust us more after reading it.
  • In-app only. The whole flow completes inside the app; never link out to a web "manage account" page (that itself would fail 5.1.1(v) and risk the Stripe-leak rejection #3).

2. Google-Gemini consent card — #196 (Apple 5.1.2(i))

What it is

This is the single most important screen you will design this quarter. It is our #1 App-Review rejection risk — the one thing most likely to get the whole app bounced on Aug 10.

Apple guideline 5.1.2(i) (added 2025, live and actively enforced since Nov 13 2025) requires: before an app shares a user's personal data with a third-party AI, it must get explicit, standalone consent that names the provider in plain text. Every message a child sends to LoLo travels to Google Gemini (via Google's Vertex AI) to generate the reply. So we must show a card that, in plain words, says "your child's messages go to Google Gemini to create LoLo's answers," and the parent must actively agree, before the child's very first message is ever sent.

The rules that make this a specific screen, not a generic toggle:

  • It must be standalone — it cannot be bundled with the microphone-permission prompt or the notification-permission prompt. Its own moment, its own decision.
  • It must name "Google Gemini" in plain text — not "our AI partner," not "a third-party service." The literal words.
  • It fires once, on a fresh install, after the COPPA parental-consent gate and before the first turn of the conversation.
  • It's a parent-context decision (the parent is setting up; the child hasn't taken over the phone yet).

Why it's the top risk: it's the newest rule, the one reviewers of AI-for-kids apps are specifically hunting, and the least battle-tested in our build. A card that fires late, is bundled with another prompt, or names the provider vaguely = automatic rejection, and rejection here is a binary fix (new build, full re-review) — the expensive kind. The founder's framing from the compliance dossier: this is "the ONE thing to fix before Aug 10."

The strings are final and approved — you are not writing copy, you're designing the moment around locked copy. The approved version is Variant B (warm-but-explicit). Its exact text:

  • Heading (EN): "LoLo needs Google's help to talk" · (DE): "LoLo braucht Googles Hilfe zum Sprechen"
  • Body (EN): "Every time {name} talks to LoLo, the message travels to Google Gemini — the AI that writes LoLo's replies — to be turned into an answer. Your child's data is stored in the EU; Google currently processes the message itself partly in the US (we're working to bring all of it to the EU). Google never uses it to train their models."
  • Consent button (single action, EN): "Yes, let LoLo talk with {name}" · (DE): "Ja, lass LoLo mit {name} sprechen"
  • Decline link (EN): "Not right now" · (DE): "Jetzt nicht"

Why Variant B and not the drier Variant A: Variant A is legally sufficient but reads like a permissions dialog a parent thumbs past — exactly the "rubber-stamp consent" pattern the Nov-2025 rule is hunting. Variant B keeps every legal fact identical (Gemini named, EU-storage/partial-US line, no-training promise, single action) but says it in LoLo's own warm voice, so a reviewer actually reads it. Your design must reinforce that — this is LoLo's calm, honest moment, not a cold legal modal.

There's also a decline path: declining does NOT delete the profile or exit onboarding. It returns to the handoff card with chat locked, and a banner explains the parent can turn on AI chat in Settings ("AI processing" row) whenever they're ready. There's no text-only fallback — Gemini generates every reply, so no consent = no chat.

How it's planned technically

  • Copy is locked (EN + DE, Variant B) — you design around it, you don't edit it. {name} is the child's first name, injected at runtime, so your layout must handle a short name and a long name gracefully.
  • Placement in the flow is fixed: onboarding (name/age → avatar → interest → handoff primer) → COPPA parental-consent card (a separate, earlier gate — that one does NOT name Gemini) → THIS 5.1.2(i) card → the disc's first turn. Your card is the last gate before LoLo speaks.
  • One decision, one primary action — a single explicit consent button plus a quiet decline link. No checkboxes to tick, no multi-step. (The COPPA card, separately, has the "I am the parent" checkbox; this one is a single tap.)
  • Bilingual — EN and DE both ship at launch; design both, and make sure the German (longer strings) doesn't overflow on a small screen.
  • The decline/locked state exists and needs a light touch: a banner on the handoff card and a re-surfacing Settings row labeled "AI processing."

Your design task (screens + states)

  1. The 5.1.2(i) consent card itself — LoLo's own calm moment. It should feel like LoLo gently explaining how she works, warm and honest, NOT a cold system permission sheet. One clear decision. The Gemini name should sit clearly in the body (it's the legally load-bearing phrase — make sure it's legible, not buried).
  2. Both languages — EN and DE layouts; verify the longer German body still breathes on the smallest supported iPhone.
  3. Small-screen safe — this must render fully (heading + full body + button + decline) without scrolling-past or truncation on a 6.1" phone. The full body text is legally required to be visible; it cannot be behind a "read more."
  4. The single consent action + the quiet decline link — clear primary ("Yes, let LoLo talk with {name}") and an unpushy secondary ("Not right now").
  5. The decline → locked state — the handoff card with chat locked + the explanatory banner ("Turn on AI chat in Settings whenever you're ready"), and the Settings "AI processing" re-entry row.
  6. Reviewer-legibility: remember a human App reviewer will see this on a fresh install and must instantly recognize it as compliant. Clarity and honesty beat cleverness here.

Hard constraints

  • Do not edit the copy. It's legally reviewed and final (Variant B). Design around it verbatim.
  • Standalone — never bundle the visual with a mic or notification prompt. It is its own full screen.
  • "Google Gemini" must be plain-text visible in the body, not abbreviated, not replaced with an icon.
  • Full body visible without scroll on small screens — no truncation, no "read more" collapse of legally-required text.
  • Light-only, warm brand, LoLo's voice. This is a trust moment, not a warning.
  • House language: this card is one of the few places we do say the mechanism plainly; keep LoLo framed as the tutor whose replies Google helps generate — not "your child's AI friend."

3. Share links + WhatsApp-preview + landing — #141 (N4)

What it is

This is the growth engine — the highest-leverage undesigned surface in the whole product. We have a $2k ad budget; we do not buy growth, we earn it through sharing. The founder's ask: when a kid makes something with LoLo (a quiz, a generated image, later a storybook), the artwork itself becomes the share — and it must look "super good."

The funnel, functionally, is a chain of five moments:

  1. Kid creates an artifact — a 5-question quiz, a generated image, later a story cover.
  2. Parent gets a share link. The parent is the sharer, not the child (kids 4–10 have no phone; the real share graph is parent → family WhatsApp group, and kid → grandparent via the parent). So the share action is a parent surface, gated behind the parent PIN/consent.
  3. The recipient opens it in WhatsApp / iMessage — and the artifact ITSELF is the preview. When the parent forwards the link into the family group chat, the message unfurls a rich preview card (the Open Graph "OG" image) where the kid's artwork is the picture, framed and wordmarked, with a title like "Mia's Dinosaur Quiz 🦕 — can you beat her score?" This is the "aunt test": the preview must instantly read "your niece made this," warm and personal — not "an app is advertising to you." The founder specifically wants this to look premium.
  4. Tapping the link opens a no-install, PLAYABLE web landing. The recipient — grandma, who will not install an app to see one quiz — lands on a fast, mobile web page where she can actually play the quiz / see the image / read the story, right there, no account, no download. Value BEFORE any install wall (the #1 thing that kills share loops is forcing an install before the recipient sees anything).
  5. A soft install CTA, then reciprocation. After the delight, a gentle "Get HeyLolo to make your own" — and, crucially, "now make one back for [Kid]." The loop only closes when the recipient becomes a creator.

Honest scope: this is a CAC amplifier, not a viral hockey stick — realistically a ~1.1–1.2× multiplier on installs we get elsewhere, plus the same artifact doubles as our App-Store screenshot and our 5-star-review moment. That's still the cheapest acquisition lever we have. Your design is the leverage — a beautiful, forwardable preview card is worth more than any ad spend.

The legal spine you must design WITHIN (non-negotiable)

Sharing a child's creation publicly is, under COPPA's 2025 rule, a "disclosure" — and disclosure of a child's personal data needs its own separate parental consent. The child tapping "share" is not consent. So the whole loop is legal only if the shared object carries no child personal information:

  • Alias only — never the child's real name on anything public. "Mia" on a shared card is a parent-approved display alias, not the child's account name. No full name, ever.
  • No child photo, no voice clip, no faceprint on the link. The shareable object is the de-identified creative output (the quiz, the generated scene, the story cover) — "look what HeyLolo made," never "look what my [named, pictured] kid made."
  • Expiring, revocable tokens. Links time out (≈30-day TTL) and the parent can kill a link anytime. Nothing lives at a permanent public URL.
  • Separate parental "share this publicly" consent — distinct from app-use consent — gates the creation of the public snapshot. The child's tap only queues it; the parent releases it.
  • No recipient tracking, no open UGC surface — no comments, no public gallery, no discovery/indexing. A link works only if you were personally sent it. The web landing collects nothing from the recipient.

This is why the design must brag about the kid (the emotional payload the parent forwards) while carrying zero actual child data. That reconciliation — pride without PII — is the central design tension of this whole feature. You resolve it with the alias + de-identified artwork approach.

The one hard technical rule that shapes your OG card

WhatsApp's preview crawler does not run JavaScript and is strict about the image:

  • OG image = 1200×630 px (1.91:1), static JPG/PNG/WebP, HTTPS, under ~400KB. An oversized image = WhatsApp silently shows no picture (the #1 failure mode). Design the card to look great and compress under 400KB at 1200×630.
  • Every artifact lives at its own unique URL — WhatsApp caches the preview per-URL for days, first crawl wins. (This is an engineering rule, not yours, but it means each shared card is one-shot: it must be right the first time.)

Your design task (screens + states)

  1. The in-app share moment — a soft, dismissible "Show this to someone? 💌" chip that appears when an artifact completes (not a modal, never blocks the next chat turn), and the parent gate it opens (parent PIN/consent) before the OS share sheet. Also the "queued for the parent" state when a child taps and no parent is present (it lands in the parent's daily digest: "Mia made a dino quiz — send it to Grandma?").
  2. The OG preview card — THIS IS A TEMPLATE, the money surface. A 1200×630 composition: [kid's artwork/quiz cover/generated image] + a branded frame + the HeyLolo wordmark, with room for the alias title ("Mia's Dinosaur Quiz 🦕"). It must:
    • work as a reusable template across artifact types (quiz cover, generated image, story cover);
    • look "super good" — premium, warm, the thing a proud parent wants to forward;
    • hold up compressed under 400KB;
    • carry the alias + artifact only, never a real name/photo. Design the frame system so any artifact drops in and it still reads as one designed card.
  3. The no-install web landing (mobile-first) — the page grandma opens. Server-rendered, fast, no login. It must be playable/readable (the quiz plays in-browser; the image/story is viewable), then end in a soft "Get HeyLolo" CTA + the reciprocation nudge ("make one back for [Kid]"). Design the quiz-play flow on web, the image/story view, and the CTA/reciprocation footer.
  4. The pre-filled share message — the parent-ready text that autofills the WhatsApp/iMessage compose ("Look what Mia made with LoLo 🦕 [link]") — you're art-directing the moment, but note the copy so it reads warm and human.
  5. The expired-link state — what a recipient sees when they open a link after its token expired or the parent revoked it. Gentle, not an error-page slap ("This one's no longer available — but you can make your own!"). A soft install CTA here too.
  6. Empty/edge: a landing for an artifact type not yet supported, and the "installed user" path (the link opens the app straight to the artifact) needs no web page but note it exists.

Hard constraints

  • Alias only, never the child's real name; no child photo/voice on any public surface. This is a legal wall, not a preference.
  • The OG card is 1200×630, static, <400KB. Design to that spec from the first pixel.
  • Parent is the sharer, parent-gated. The child never publishes alone.
  • No recipient tracking, no comments, no public gallery/discovery. The landing collects nothing.
  • Light-only, warm brand. The card and landing carry the HeyLolo look off-app for the first time — this is our brand meeting strangers, so it must be impeccable.
  • House language on the landing + CTA: "safe AI tutor for children," never "AI friend/companion." A stranger's first impression of HeyLolo is a parent-trust impression.
  • No child-visible pay pressure anywhere in the loop; the install CTA is aimed at the recipient adult, not the child.

5. Wonder Shelf reframe — #142 (N5)

What it is

The founder wants "badges to unlock and collect." The gamification research says: for under-10s, collections beat points and badges — because a collection is completeable, story-tied, and permanent, which is what actually satisfies a young child (identity + "I did it" + a natural endpoint). So we ship the founder's intent (collect + unlock) in the correct shape: the Wonder Shelf.

The Wonder Shelf is a shelf/museum the kid curates, not a scoreboard. Every real accomplishment — a conversation milestone, a completed quiz, a finished story, a generated image, an Explorer discovery, a mood check-in — leaves a small collectible artifact on a permanent shelf: a fossil, a shell, a candle, a ribbon, a little glowing token. Artifacts group into collections of 6–10 (e.g. "Ocean Deep": sea shell, pearl, coral, jellyfish, treasure coin, diver's mask, whale song, and a golden capstone). Collections complete ("4 of 6 ocean creatures found"), and completing one closes an arc — LoLo celebrates once. There are 12 launch collections, ~102 artifacts total (4 always-on starter sets, 4 interest sets tied to the child's onboarding interest, 4 seasonal sets tied to the Friday-drop calendar).

The critical reframe: the existing badges-baseline.html is the wrong shape — it's a badge scoreboard. The Wonder Shelf is the opposite: a warm, curated shelf/museum where nothing is ever lost or decays. No streaks. No XP. No numbers shown to the child. No leaderboard. And this is not aesthetic preference — it's a hard product/legal stance you must design inside, so please internalize the reasoning below so you don't accidentally reintroduce engagement pressure.

Why NO streaks / XP / leaderboard (the reasoning, so you don't design them back in)

  • The EU Digital Services Act (DSA) minors guidelines name streaks and random-reward mechanics as things that must be off by default for minors. Building them in is a regulatory liability, not just a taste call.
  • The science: for under-10s, points/XP/leaderboards are developmentally weak-to-harmful — kids can't do normative comparison yet, streaks create documented anxiety (kids don't control their own schedule, so a lost streak is a punishment for something outside their control), and random rewards correlate with problem-gambling patterns more strongly in minors.
  • The cautionary tale is Prodigy — a math app that wrapped learning in a reward economy with pay-to-progress cosmetics; it drew an FTC complaint and the press image of "membership kids sail on a cloud while non-members trudge in the dirt." That is the exact dark-pattern failure the Wonder Shelf is designed to be the opposite of. Even Duolingo removed streaks/leagues/leaderboards for its own kids app (Duolingo ABC) — the best-funded team's revealed preference.
  • So the anti-list is hard: no numbers/XP shown to the child, no streak calendar, no leaderboard, no mystery boxes/wheel spins, no child-visible padlocks that flaunt a purchase. Progress is shown visually — filled vs empty slot silhouettes on the shelf — never "12/50." Loss-free by construction: artifacts are permanent, sets never expire, a missed seasonal set re-runs next year (nothing permanently missable).

The positive design DNA to lean into instead: permanence (the shelf only ever grows), completeable collections (6–10, with a golden capstone that auto-mints when the set fills), the mint-moment (when an artifact is earned it visibly flies to the shelf — the permanence beat), LoLo's own stake (LoLo can have her own corner of the shelf that fills alongside the child's — the Khan Kids "collect for the character" shape), and celebrate once (single artifact = a tiny whisper tick; set completion = one landmark celebration, never back-to-back confetti).

How it's planned technically

  • Artifacts are a projection over data that already exists — messages, generated images, finished stories, quiz completions, Explorer captures, mood docs. The shelf doesn't invent new tracking; it maps real events to artifact slots via deterministic, idempotent earn rules (a fixed event → a fixed artifact, no randomness).
  • 12 launch collections ship bundled in-app as PNGs; post-launch seasonal/interest sets arrive via the Friday-drop system from Firebase Storage (so the shelf can grow without an app update). ~102 artifact images, generated via the existing Imagen 4 Fast pipeline with a style-lock prompt (same object-swap trick the story pipeline uses for consistency) — and you curate them (generate 2–3× overage per slot, pick the best for style/palette/kid-safety/single-object clarity at small shelf size).
  • Everything sits behind the existing gamification parent toggle (conservative default; off = surface hidden, engine still records silently). Earned-while-away lands only in the parent digest, never as a push to the child.
  • Natural home: the chats-history area / a shelf surface. Progress counters exist backend-side only, never rendered to the child.

Your design task (screens + states)

  1. Shelf home — the museum/shelf metaphor. All collections at a glance, showing filled vs empty slots visually (silhouettes), progress legible without numbers. This is the surface that replaces the scoreboard badges-baseline — establish the shelf/curation feeling.
  2. Collection detail — one collection (e.g. "Ocean Deep"): its 6–10 slots, filled artifacts vs empty silhouettes, the golden-capstone slot, and the "closed arc" state when complete. Show how "4 of 6" reads visually, no digits.
  3. Artifact detail — tap one artifact: what it is, and how it was earned ("You found this when you asked LoLo about whales"). The earning story is the emotional payoff — permanence + memory.
  4. The "new artifact lands" moment — the mint animation: an artifact flies to the shelf in-session (the permanence beat). Two tiers: a small whisper tick for a single artifact; the one landmark celebration when a set completes (LoLo celebrates once). Design both, and the pacing guard feeling (never two big celebrations in a row).
  5. Empty / first-run state — the shelf before anything's earned. It must be inviting, not a wall of locked padlocks (no padlocks that flaunt purchase). Consider the Day-0 guarantee: within the first ~3 minutes the child earns their first two artifacts (Hello Spark, Mood Pebble) — design so the empty shelf clearly wants to be filled and the first mint feels magical.
  6. LoLo's corner (optional but wanted) — LoLo's own shelf area that fills as the child's does (the generosity/character-stake loop).
  7. Seasonal-set integration — how a seasonal collection (Autumn, Halloween, St. Martin's Lantern, Advent) appears and re-runs annually without anything feeling "missed."
  8. Art-direction pass on the ~102 artifacts — you own the curation/QC of the Imagen-generated tokens (style/palette consistency, kid-safety, single-object clarity at ~96px). This is a real chunk of the work (~a designer-day).

Hard constraints

  • NO streaks, NO XP, NO numbers shown to the child, NO leaderboard, NO mystery boxes/wheel, NO child-visible padlocks-with-price. This is the DSA/dark-pattern anti-list — design pressure is a legal and brand liability, not a taste debate.
  • Loss-free + permanent. Artifacts never disappear; sets never expire; missed seasonal sets re-run. Nothing decays.
  • Celebrate once. Single artifact = whisper; set complete = one landmark. No confetti spam.
  • The badges-baseline is the wrong shape — do not iterate on the scoreboard; design the shelf/museum fresh.
  • Deterministic earn — every artifact has a fixed event trigger; no randomness, ever (DSA).
  • No child-facing paywall pressure — the 12 launch sets are all earnable by any child on any plan; if a set is ever Plus-only, a non-Plus child simply never sees it exists (no teasing padlock). Sell breadth of sets to the parent on the paywall, never to the child on the shelf.
  • Light-only, warm tide-pool palette. The artifact style-lock is warm teal/coral/sand/cream, soft rounded chunky shapes, matte storybook illustration.

6. Friday Drop arrival moment — #152 (D1) + #154 (D3) + #121

What it is

Every Friday, a free quiz deck appears for everyone. It's the retention ritual — the "appointment mechanic" that gives a child a reason to come back, the way a Toca Boca kid runs to the in-game Post Office every Friday for the free gift, or a Moshi kid gets a new sleep story weekly. Because a quiz deck is pure LLM text, it costs ~$0 to "press" and serve to the whole user base — this is the AI-native superpower (Tonies pays €15 per physical figurine; we press a "figurine" for nothing).

The ritual is the point — predictable anticipation. Kids know something fresh lands Friday, they talk about it, it becomes a shared beat. The content is already authored: 13 themed decks covering the Sep–Dec run (issue #154 frames it as "16" — see the reconciliation below; both are correct), starting Sep 18 (back-to-school), running through autumn, Halloween, St. Martin's/Laternenfest (a big DACH beat), Nikolaus, and Christmas.

The hard discipline that makes this legal and on-brand: it is a deterministic, loss-free drop. "A new deck appeared this Friday" — scheduled, everyone gets it, never a mystery box, never a streak you can lose, and NO push notification to the child. Discovery happens in-app (a badge/shimmer glows on the quiz surface) plus one line in the parent digest. Old decks are never deleted — they move to a browsable "last week's decks" library/shelf. This is the DSA-safe translation of the live-ops appointment mechanic: all the return-habit upside, none of the regulated dark-pattern (no streak-loss, no random reward, no child-push).

Your job is the arrival UX — how a new deck announces itself without becoming a nag or a takeover. It should feel like a small gift quietly appearing, discovered with delight, not an interruption demanding attention.

The 13-vs-16 reconciliation (I dug into this so you're not confused)

  • Issue #154 says "16 themed decks." The content-drop-economy doc lays out a 16-week Sep–Dec calendar. But of those 16 weeks: week 1 is launch week (no drop), and week 13 folds into the Advent Calendar (no standalone quiz). That leaves 14 distinct weekly Friday quiz themes. One of those ("Animal Kingdom") already shipped as an example in the authoring spec. So the companion doc authors the remaining 13 decks. Net: ~13–14 weekly Friday decks for Sep–Dec, all authored and ready. Don't let the "16" vs "13" throw you — it's the same run counted two ways (calendar-weeks vs decks-still-to-author).

How it's planned technically

  • Backend flips a deck live on its scheduled Friday — a server-scheduled catalog of dated content (drops collection: id, type, theme, publishAt, featuredUntil, tier, asset refs). When publishAt passes, the deck goes live and the previous one moves to the library. Zero per-week ops — the whole Sep–Dec run is batch-authored and scheduled once.
  • The surface ships in the binary; the content is server-schedulable — so the arrival UX you design must ship in a v1.1 build, but decks themselves flip live without an app update.
  • It integrates with the Wonder Shelf (#142) — opening a Friday drop mints a "Friday Gift Box" artifact + a seasonal-collection slot, and the "last week's decks" library is a shelf-adjacent browse surface. Coordinate the visual language with feature 5.
  • The quiz engine (#121) runs the deck itself (already built, mode:"quiz"); the Friday Drop is the scheduling + arrival + discovery layer on top.
  • Parent-digest slot — the "come back" nudge goes to the parent (opt-in digest), never a push to the child: e.g. "This Friday's deck: Lanterns & Light 🏮 — Mia loved autumn week."

Your design task (screens + states)

  1. The in-app arrival — a badge/shimmer on the quiz surface signaling "something fresh landed," discovered in-app. NOT a modal takeover — no full-screen interrupt, no "tap to claim!" nag. A gentle glow/sparkle that invites, doesn't demand. Design the "new" state and how it settles once the child has opened it.
  2. The deck-reveal moment — opening the new deck: a small, warm "here's this Friday's deck" reveal (the gift-box/unwrap feeling, tied to the Wonder Shelf "Friday Gift Box" artifact mint). Celebratory but light — one whisper/moment tier, never confetti overload.
  3. The "last week's decks" shelf/library — where past drops live (never deleted). How a child browses previous decks; how it reads as "always available, this one's just the newest." Integrate visually with the Wonder Shelf (#5) so it feels like one world.
  4. The parent-digest slot — the one-line "fresh this Friday" mention in the parent's digest/dashboard (the only place a "come back" nudge is allowed to live). A small, warm card/line — "This Friday: [deck theme]."
  5. Seasonal theming — the drops are themed (back-to-school, autumn, Halloween, Laternenfest, Nikolaus, Christmas). Show how the arrival surface picks up the season's mood without needing a bespoke design each week (a themeable frame, like feature 3's card system).
  6. Empty/between state — the quiz surface when there's no new drop (mid-week), so "new" genuinely means new. And the launch-week state before the first drop (Sep 18).

Hard constraints

  • NO push notification to the child. Discovery is in-app + parent digest only. (COPPA separate-consent + DSA + brand.)
  • NOT a modal takeover. The arrival is a badge/shimmer the child discovers, never a full-screen interrupt that blocks play.
  • Deterministic + loss-free. "A new deck appeared" — never a mystery box, never a streak, never FOMO-punishment. Old decks stay in the library, never deleted. Soft "featured this week" is the only freshness edge, and even that never punishes.
  • No numbers/XP/streak on the child surface (same anti-list as feature 5).
  • Integrates with the Wonder Shelf — shared visual language, the Friday Gift Box artifact.
  • Light-only, warm brand. House language in any parent-facing copy: "safe AI tutor," never "companion/friend."
  • No child-visible pay pressure — the Friday quiz decks are free for everyone, deliberately (gating them would kill the ritual). Never a padlock on a Friday deck.
