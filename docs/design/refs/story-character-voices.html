Story character voices — research dossier

Question: For the story-creation feature (stories read aloud to kids 4–12), can HeyLolo give story characters distinct voices — either (a) multiple TTS voices per story (speaker switching) or (b) one voice that ACTS each character differently (prompted character styling)?

Date: 2026-07-16 · Stack: Gemini TTS, gemini-3.1-flash-tts-preview, voice "Zephyr", child-only v1_synthesizeSpeech. Scope note: this is about in-story character voices, NOT LoLo persona packs (docs/research/voice-persona-packs.md — a separate feature; do not conflate).

TL;DR recommendation: Ship (b) narrator = Zephyr + style-prompted acting first — it's a prompt/annotation change only, zero FE change, zero cost delta, zero latency delta, zero compliance change. Native multi-speaker is a dead end for this use case (hard cap of 2 speakers; a story has narrator + several characters). If acting quality proves insufficient, upgrade to per-speaker-run multi-voice chunking (unlimited voices, some seam risk) using the same LLM annotations — never native multiSpeakerMarkup.

1. The real backend contract (path:line — map recommendations onto this)

The TTS path is single-voice, streaming, one WAV per assistant message. There is no story/segment concept today.

  • Provider interface — elumi-kids-api/functions/src/api/text-to-speech/providers/tts-provider.ts:9-28
    • synthesize(text, { languageCode?, voice?, onAudioChunk? }). Every driver streams raw PCM16-LE mono @24kHz; swapping providers needs zero FE change.
  • Gemini provider — .../providers/gemini.provider.ts
    • DEFAULT_GEMINI_MODEL = "gemini-3.1-flash-tts-preview" (:17); GA fallback gemini-2.5-flash-tts (:24).
    • The style prompt is ALREADY wired: const prompt = voicePreference?.style ?? DEFAULT_GEMINI_PROMPT (:77), default "Read aloud in a warm, welcoming tone." (:30).
    • Streaming call (synthesizeStreaming, :119-202): one IVoiceSelectionParams voice (name, languageCode, modelName, :79-83) written to streamingConfig.voice; text+prompt ride the input chunk: stream.write({ input: { text, prompt } }) (:134). This is the single-speaker shape — no multiSpeakerVoiceConfig.
    • Latency budget: FIRST_CHUNK_TIMEOUT_MS = 4_000 (:47), STREAM_INACTIVITY_TIMEOUT_MS = 15_000 (:48). Any character-voice scheme must keep first audio < 4 s and never gap > 15 s, or the watchdog kills the stream.
  • Orchestrator — .../text-to-speech.service.ts
    • TTS_MAX_INPUT_CHARS = 4_000 (:35), clamped via truncateAtSentenceBoundary (:90). A 1000-word story ≈ 5–6.5k chars > 4000 — so long stories ALREADY require paragraph chunking, regardless of the voice approach. This is the single most important architectural implication: chunking is unavoidable, which makes per-chunk voice/style switching natural, not extra cost.
    • Output is PCM16 mono @24kHz → wrapPcm16AsWav (:16-18, :140).
  • Voice preference — .../child/child.entity.ts:140-153
    • TtsVoicePreference = { provider, voiceId, style?: string | null, model?: string | null }. style already carries a steering prompt to Gemini (:147-149). Chirp has no style channel.
  • Callable — .../synthesized-speech/synthesized-speech.service.ts:61-154
    • One WAV per {childId}:{messageId}; voice resolved server-side from child.ttsVoice (:118); streaming path returns url: "" and the FE plays the streamed chunks (:133-145).

Takeaway: approach (b) is a content change (enrich text/style) that rides the existing contract untouched. Approach (a) requires either the native 2-speaker markup (doesn't fit) or a new per-run chunking loop in the service — but the FE, being PCM-uniform, still needs no change as long as chunks stream in order.

Verdict on native multi-speaker: The 2-speaker cap disqualifies it for a typical story (narrator + 3 characters = 4 voices). It could only do narrator + one character per request. To give ≥3 characters distinct voices you must fall back to per-speaker-run chunking (§4b) anyway — at which point the native markup buys you nothing over just switching voice.name per chunk. So: skip multiSpeakerMarkup.

3. Style / acting control on ONE voice (Zephyr) — this is the viable lever

Gemini-TTS is explicitly built for prompt-steered delivery, and the prompt field is already plumbed in our provider (gemini.provider.ts:77,134).

  • Natural-language steering of "style, accent, pace, tone, and emotional expression." Zephyr (one of the 30 prebuilt voices) is a first-class steerable voice. Source: blog.google — Improving Gemini TTS, docs.cloud.google.com/text-to-speech/docs/gemini-tts.
  • Inline markup tags (Preview, on 3.1) — 4 tag families that apply to subsequent speech, so you can switch acting mid-utterance within one request:
    • Non-speech sounds: [sigh] [laughing] [uhm]
    • Style modifiers: [sarcasm] [robotic] [shouting] [whispering] [extremely fast]
    • Vocalized adjectives: [scared] [curious] [bored]
    • Pacing: [short pause] [medium pause] [long pause]
    • 3.1 Flash TTS advertises 200+ audio tags. Sources: chatforest Gemini 3.1 Flash TTS review, Cloud docs above.
  • Scope: the top-level prompt is GLOBAL to the synthesis input; inline [tags] are local to the speech that follows them. So a single Zephyr stream can voice a grumpy bear line [shouting][annoyed] "Who ate my porridge?!" then a timid mouse [whispering][scared] "not me…" — one voice acting two characters in one streaming request. This is exactly the "(b) one voice that acts" ask.
  • Reliability caveat (be honest): Google documents "voice inconsistency with prompt instructions" — the model doesn't always strictly obey. Source: ai.google.dev/gemini-api/docs/speech-generation. Practical read: great for flavor (a whisper, an excited beat, a slow spooky line), less reliable as a guaranteed per-character timbre. If two characters must sound unmistakably different every time, that's the argument for multi-voice (§4b).

4. Approaches, mapped to our contract

(b) Narrator = Zephyr + style-prompted acting — RECOMMENDED FIRST

  • How: story LLM emits the text with inline audio tags at dialogue boundaries; BE sends it through the existing single-voice stream. voice.style (or the per-chunk prompt) sets the narrator's baseline tone.
  • Contract change: none. text gets richer; prompt optionally per-chunk. FE: none.
  • Cost / latency: identical to today (tags are a few input tokens; §5).
  • Limit: one timbre (Zephyr) acting all roles — differentiation is performance, not distinct voices; subject to the consistency caveat.

(a) Multi-voice via per-speaker-run chunking — UPGRADE PATH (only if (b) falls short)

  • How: LLM tags each segment with a speaker; BE maps speakers → prebuilt voices (Zephyr narrator + a small palette, §6); each contiguous same-voice run is one streaming request; PCM is concatenated into the queue in order. Unlimited distinct voices; sidesteps the 2-speaker cap.
  • Contract change: new chunk loop in text-to-speech.service.ts + a speaker→voice map. FE: none (PCM stays uniform and in-order).
  • Cost: same total (cost is audio-duration-driven, §5).
  • Risk: seams — prosody/loudness discontinuities between runs (each request is independently normalized), and inter-request latency: a new stream's first chunk must arrive before the queue drains. Mitigate with lead/prefetch (synthesize run N+1 while run N plays) — the codebase already leans on deep buffering (docs/_session/perf-architecture-deep-dive/). Must respect FIRST_CHUNK_TIMEOUT_MS.

native multiSpeakerMarkup — REJECTED

  • 2-speaker cap can't express narrator + ≥2 characters; no cost/latency advantage over (a). Skip.

5. Cost & latency model (typical bedtime story ≈ 1000 words, narrator + 3 characters)

Cost is dominated by AUDIO OUTPUT and is essentially identical across all three Gemini approaches — the per-token rate has no multi-speaker premium, and total spoken audio duration is the same however you slice it.

  • Output audio tokenized at 25 tokens/sec (blog / pricing, tokencost.app/blog/tts-api-pricing-2026).
  • 1000 words ≈ ~6.5 min speech ≈ ~390 s × 25 = ~9,750 output tokens → at $10 / 1M (Gemini 2.5 Flash TTS) ≈ $0.10/story.
  • Input: ~6k chars ≈ ~1.7k tokens (+ style prompt/tags, tens of tokens) → at $0.50 / 1M ≈ $0.001 — negligible.
  • Single-voice vs style-prompted vs multi-voice = same ~$0.10. Style tags add a rounding-error of input tokens; chunking pays the same total output. Cost is NOT a differentiator — quality, latency, and effort are. Rate source: tokencost.app/blog/tts-api-pricing-2026, cloud.google.com/text-to-speech/pricing.

Latency: streaming is supported on gemini-3.1-flash-tts-preview (our model). (b) is byte-for-byte the current path → no change. (a) adds a first-chunk gap each time the voice switches; keep the queue fed by pre-synthesizing the next run during playback and by making runs paragraph-sized (few switches). Both must stay inside the existing 4 s first-chunk / 15 s inactivity watchdog (gemini.provider.ts:47-48).

6. Voice palette (if/when we do multi-voice)

30 prebuilt Gemini TTS voices (all Preview on 3.1). Kid-story-suitable picks alongside Zephyr (narrator, the founder's pick):

  • Warm/female: Aoede, Callirrhoe, Leda, Sulafat, Achernar, Despina — gentle characters, mothers, kind animals.
  • Bright/playful: Puck, Laomedeia, Pulcherrima — sprightly sidekicks, small creatures.
  • Deep/authoritative male: Charon, Enceladus, Orus, Algieba — giants, gruff bears, wise elders.
  • Neutral/clear: Kore, Iapetus, Schedar — narrator alternates, robots.

Source (voice list): docs.cloud.google.com/text-to-speech/docs/list-voices-and-types, chatforest review. Seam caveat: mixing voices across chunked streaming requests can show loudness/prosody discontinuities (each request self-normalizes). Test 2–3 palette voices against Zephyr for level-matching before committing.

7. Alternatives (only if Gemini falls short — one paragraph each)

  • Google Cloud TTS Chirp 3 HD / Studio — already a provider in-tree (chirp.provider.ts, the no-vendor fallback). No style/steering channel (our own code comment: "Chirp 3 HD has no style channel", child.entity.ts:147-149) and no multi-speaker acting — strictly a fallback timbre, not a character-voice engine. No compliance change (Google).
  • OpenAI gpt-4o-mini-tts — strong natural-language steerability (instructions param, 13 voices), often praised as more obedient than Gemini for "vibe" control. **$0.60/1M input + $12/1M audio out (~$0.015/min)** ≈ ~2× our Gemini output rate; 2000-token input cap forces chunking too. Sources: developers.openai.com/api/docs/models/gpt-4o-mini-tts, blog.promptlayer.com/gpt-4o-mini-tts. Compliance flag: OpenAI is a NEW subprocessor — a COPPA data-flow/DPA review is required before any kid text is sent. It's text-in/audio-out (no child-voice capture), so lower risk than ElevenLabs, but still a vendor-change decision, not an engineering default. Flag, don't decide.
  • ElevenLabs — best-in-class character acting, but COPPA-hostile posture: services "not intended for or directed at children under 18," child/child-like voices banned, and voice samples "may be stored and used to improve our services" with no guaranteed deletion; commentary flags that self-serve doesn't enforce verifiable-parental-consent for the biometric/voiceprint data COPPA now covers. Source: elevenlabs.io/privacy-policy, elevenlabs.io/use-policy, COPPA 2025 guide — promise.legal. Our data-honesty canon: processing already goes through Google; ElevenLabs is a new subprocessor with documented children's-data concerns → HIGH compliance risk. Recommend NOT adopting for a kids' product.

8. Implementation sketch (recommended path (b), upgrade-ready to (a))

  1. LLM annotation format (do this now, serves both approaches). Have the story-generation model emit structured segments, e.g.:
    { "segments": [
  { "speaker": "narrator", "text": "The forest was quiet.", "emotion": "calm" },
  { "speaker": "bear", "text": "Who ate my porridge?!", "emotion": "angry" },
  { "speaker": "mouse", "text": "Not me…", "emotion": "scared" }
] }
    speaker is the field that later unlocks multi-voice; emotion maps to audio tags now.
  2. Render for (b) — single Zephyr stream, no contract change. BE flattens segments into one tagged text run: narrator plain; dialogue prefixed with the emotion tag → The forest was quiet. [angry][shouting] "Who ate my porridge?!" [whispering][scared] "Not me…". Send through the existing synthesize() path; style/ prompt sets the narrator baseline. FE untouched.
  3. Chunk long stories. Because TTS_MAX_INPUT_CHARS = 4000 (text-to-speech.service.ts:35), split the story at paragraph/segment boundaries into ≤4000-byte streaming requests and pre-synthesize the next chunk during playback so the queue never starves. This is required for stories regardless of the voice approach — build it once.
  4. Upgrade to (a) later, same annotations. Add a speaker → voiceId map; render each contiguous same-speaker run as its own streaming request (switch IVoiceSelectionParams.name); concatenate PCM in order. Reuse the chunk/prefetch machinery from step 3. Level-match palette voices to Zephyr first (§6 seam caveat). Still no FE change (PCM stays uniform and in-order).

9. Open questions

  1. Story length policy — do we cap read-aloud stories, or fully chunk arbitrarily long ones? (Drives the prefetch depth and the 4000-byte chunker.)
  2. Acting consistency threshold — is Zephyr-acts-all "good enough," or do parents/kids expect distinct voices? This decides whether we ever build (a). Needs the A/B below.
  3. Preview-model dependency — inline audio tags + streaming are Preview features on gemini-3.1-flash-tts-preview. GA gemini-2.5-flash-tts (our fallback) streams but tag support is thinner — confirm degraded-but-acceptable acting on the GA fallback path (gemini.provider.ts:24,100-116).
  4. Palette level-matching — do Aoede/Charon/Puck sit at the same loudness/prosody as Zephyr across chunk boundaries, or do we need per-voice gain normalization?
  5. Caching — one WAV per {childId}:{messageId} today. Stories are long + regenerated; decide cache granularity (per-story vs per-chunk) to avoid re-paying synthesis on replay.

10. Suggested device-testable A/B (same story paragraph, 3 renders)

Take one ~120-word paragraph with narrator + bear + mouse dialogue and synthesize it three ways, play on device back-to-back, and have the founder + a kid rate "can you tell the characters apart / does it sound good":

  • A — plain single voice (today): Zephyr, default warm prompt, no tags. Baseline.
  • B — style-prompted acting (recommended): Zephyr + inline [angry][shouting] / [whispering][scared] tags at dialogue. One request, no FE change.
  • C — multi-voice chunked: Zephyr narrator + Charon (bear) + Puck (mouse), one streaming request per run, concatenated. Listen for seams/loudness jumps and any gap between voice switches.

Decision rule: if B is convincingly expressive, ship B and defer C. If characters blur together in B, C justifies the extra BE work — and the LLM annotations from step 1 make C a rendering swap, not a redesign.
